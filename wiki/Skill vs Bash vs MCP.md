# Skill vs Bash vs MCP

LLM Agent 的三種工具模式，各有不同的定位與取捨。正確選擇的前提是理解 LLM 的本質——機率模型，不是智能體。

## 前提：LLM 是吃角子老虎機

LLM 根據訓練資料預測下一個字的機率。它不「理解」你的指令，而是根據輸入的機率分布生成最可能的輸出。所有工具選擇都應基於這個認知。

## 比較

| 維度 | Skill | Bash | MCP |
|------|-------|------|-----|
| 本質 | 引導機率分布 | 操控 CLI 工具 | 固定接口工具 |
| 目的 | 提高 LLM 生成預期結果的機率 | 讓 LLM 執行系統操作 | 讓 LLM 安全地呼叫預定義功能 |
| 效率 | 高（純 prompt） | 高（可串聯多工具） | 中（需包裝） |
| 安全性 | 中（受限於 LLM 能力） | 低（可能破壞檔案系統） | 高（格式受限、可擋錯） |
| 適用邊界 | LLM 訓練資料涵蓋的範圍內 | 已有 CLI 工具可用時 | 需要精確、可驗證的操作時 |
| 建置成本 | 低（寫 prompt） | 低（寫指令） | 高（需開發接口） |

## 何時用什麼

### 用 Skill
- LLM 訓練資料中已涵蓋該能力
- 需要引導輸出格式、分析方法、報告結構
- 快速迭代、一天內可完成

### 用 Bash
- 已有成熟 CLI 工具
- LLM 只需生成簡單的串接指令
- 權限控管與版本控制已到位
- 可接受一定程度的風險

### 用 MCP
- 需要精確、可驗證的操作結果
- 涉及 LLM 訓練資料外的特殊演算法
- 安全性要求高，不能容忍破壞性操作
- 需要穩定的輸入輸出格式

## 風險光譜

```
安全 ←———————————————————→ 危險
  MCP          Skill          Bash
（受限接口）  （機率引導）  （開放執行）
```

## 核心洞見

- **Skill 不是教 LLM 怎麼做**，而是提高勝率
- **Bash 的風險**來自 LLM 生成指令的機率性——一行 `rm -rf .` 就能毀掉一切
- **MCP 的價值**在於把「已驗證的正確邏輯」包成工具，LLM 只負責呼叫
- 三者不互斥，實務上常組合使用
