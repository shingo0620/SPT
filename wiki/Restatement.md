# Restatement（重申機制）

在 LLM Agent 的長任務執行中，週期性地將關鍵資訊重新推回模型注意力最強的位置，確保規則和任務不被累積的上下文「埋掉」。

## 為什麼需要 Restatement

LLM 對上下文不同位置的注意力不均等：

```
注意力強度
  ▲
  │ ████                                    ████
  │ ████                                    ████
  │ ████                                    ████
  │ ████  ░░░░░░░░░░░░░░░░░░░░░░░░░░░░░   ████
  │ ████  ░░░░░░░░░░░░░░░░░░░░░░░░░░░░░   ████
  └──────────────────────────────────────────────→
   System    中段（大量累積內容）        最新內容
   Prompt                              （尾端）
```

- **注意力最強**：System Prompt（開頭）+ 最新生成的內容（尾端）
- **注意力最弱**：中段——上下文越長，中段越容易被忽略

當 Agent 執行到第 100 步時，第 1 步注入的 Skill 和規則雖然還在上下文中，但模型**不再優先注意**。這不是「遺忘」，而是注意力被稀釋。

## Restatement 的兩大放置位置

### System Prompt（靜態規則）

適合放入**對所有任務都適用、不會變動**的規則：
- 代碼風格
- 輸出格式要求
- 全域禁止行為（如「不可合併步驟」）

**優點**：始終處於注意力高地，KV Cache 可複用
**注意**：一旦修改任何一個字，從修改位置起的 KV Cache 全部失效、需重算

### 上下文尾端（動態資訊）

適合放入**隨執行進度變化**的資訊：
- TodoList 進度（做到哪一步、下一步是什麼）
- 當前階段的 Skill 規則重申（**下一步怎麼做**）
- 累積的 Findings 和 Plan

**優點**：不破壞 System Prompt 的 KV Cache，且天然處於注意力高位
**做法**：在每次 Agent 輪次之間，將重申內容追加到上下文最尾端

## KV Cache 的實務影響

```
[System Prompt] [歷史對話 1] [歷史對話 2] ... [最新輸入]
      ▲                                          ▲
  修改這裡 → 後面全部重算 💸         追加這裡 → 只算增量 ✅
```

| 操作 | KV Cache 影響 | 成本 |
|------|--------------|------|
| 追加到尾端 | 前面全部複用，只算新增部分 | 低 |
| 修改 System Prompt | 從修改位置起全部重算 | 高 |
| 修改中段內容 | 從修改位置起全部重算 | 高 |

**實務建議**：動態 Restatement 永遠追加到尾端，不要嘗試「更新」System Prompt 中的動態欄位。

## Restatement 的完整內容清單

僅重申「做到哪一步」是不夠的。完整的 Restatement 應包含：

| 內容 | 回答的問題 | 解決的症狀 |
|------|-----------|-----------|
| TodoList 進度 | 現在在哪？下一步是什麼？ | 趕工（跳步/合併） |
| Skill 規則 | 下一步**怎麼做**？ | 規則失效 |
| Findings | 前面發現了什麼？ | 資訊遺失 |
| Plan 概要 | 整體方向是什麼？ | 偏離目標 |

> 在長任務裡面，適當地重複、適當地廢話，不是浪費，而是必要的控制手段。
> ——数字黑魔法

## 與上下文隔離的搭配

Restatement 解決「該看到的資訊沒被看到」。
但有時問題相反——「不該看到的資訊被看到了」（如前面的代碼導致後面的輸出同質化）。

此時需要 **Sub-Agent 做上下文隔離**：
- 開啟乾淨的上下文窗口
- 只注入必要的 Restatement 內容
- 屏蔽不需要的歷史輸出

兩者天然互補：Restatement 決定「要帶什麼進新窗口」，Sub-Agent 提供「乾淨的窗口」。

## 與 [[AI 輔助軟體工程]] 的呼應

Matteo Barbero 方法中「每個 task 開全新 session」的做法，本質上就是一種**極端的上下文隔離**——每次都給 Agent 完全乾淨的上下文，透過結構化文件（PRD → Issues → Tasks）做 Restatement。

| 做法 | Restatement 方式 | 隔離程度 |
|------|-----------------|---------|
| 全新 session | 結構化文件注入 | 最高（完全乾淨） |
| Sub-Agent | 選擇性上下文注入 | 中（保留必要脈絡） |
| 同一 session 追加 | 尾端追加重申 | 最低（共享所有歷史） |
