# Claude Code 五個底層概念

> 來源：[AgentCrew Academy — YouTube](https://www.youtube.com/watch?v=srS_8Y8hxd0)（2026-04-15）
> 講者：Dustin

## 概述

五個理解 Claude Code（及所有 LLM Agent）行為的核心心智模型：**概率、窗口、模型、循環、層級**。理解這些概念後，能更有效地駕馭 AI 協作。

---

## 1. 概率（Probability）

LLM 的產出是基於**下一個 token 的機率預測**。

- **例子**：前面的字是「白日」→ 如果上下文偏文學，「依山盡」機率高；如果偏近代史，「滿地紅」機率高
- **核心認知**：LLM 不是「理解」，是「預測」。隨著模型進步錯誤越來越少，但**永遠有出錯的可能**
- **實務要點**：
  - 模型知識停留在訓練完成的那一刻（Knowledge Cutoff）
  - 有時效性或必須正確的資訊（軟體版本、法律規範），**必須讓模型上網搜尋**，不能只靠訓練資料

## 2. 窗口（Context Window）

上下文窗口是**稀缺資源**，管理好壞直接影響模型表現。

- **LLM 是無狀態的（Stateless）**：每次請求都不會「記得」之前的對話。背後的程式把所有歷史對話打包一起送出，才產生「記憶」的錯覺
- **上下文缺失** → 忘記重要資訊
- **上下文過多** → 注意力稀疏（attention dilution），模型選擇性或隨機遺忘某些內容
- **管理手段**：
  - `/clear` — 完成一個任務後清除上下文，釋放空間
  - `/compact` — 任務進行中但窗口快滿時，壓縮歷史為摘要
  - CLAUDE.md / 知識庫 — 讓模型「在對的時間讀到恰好對的資訊」，而非一股腦塞入

> 這正是 [[Restatement]] 要解決的問題——在長任務中週期性重申關鍵資訊，避免注意力衰減。

## 3. 模型（Model Tiers）

Claude 有三個等級，各有適用場景：

| 模型 | 定位 | 適用場景 | 額度耗費 |
|------|------|---------|---------|
| **Opus** | 最聰明 | 解 bug、大型系統重構、複雜學術問題 | 最高 |
| **Sonnet** | 主力 | 方向確定後的執行、日常任務 | 中等 |
| **Haiku** | 輕快 | 快速搜尋、文檔總結、小型 Subagent | 最低 |

- Sonnet 是日常主力模型
- Haiku 適合設定給小型 Subagent 處理簡單任務

## 4. 循環（Agent Loop）

Claude Code 與一般聊天 AI 的根本差異。

- **聊天 AI**：你問一句、它答一句，結束
- **Agent Loop**：自主調用工具 → 讀取結果 → 決定下一步 → 失敗時修正方向 → 繼續

> 「做一步、看結果、決定下一步」的循環就是 Agent Loop。這就是為什麼它能連續處理複雜的多步驟任務，真的「幫忙做事」而不只是「回答問題」。

這與 [[AI 輔助軟體工程]] 中描述的 Agent 工作模式一致——LLM 不只生成文字，而是在 loop 中操作工具、自我修正。

## 5. 層級（Prompt Injection Hierarchy）

你每次發送一則訊息，實際上前面包了多層系統訊息。**層級越高，優先級越高**。

| 層級 | 名稱 | 說明 | 可修改？ |
|------|------|------|---------|
| 1（最高） | System Prompt | Anthropic 內建的行為規範（不能做有害的事） | 不可 |
| 2 | CLAUDE.md + Rules | 你與 AI 的「默契檔」——專案規範、協作方式 | 可 |
| 3 | 對話訊息 | 你每次跟 Claude 說的話 | 可 |
| 4（最低） | 工具回傳結果 | 調用工具後拿到的資訊 | 不可（由工具決定） |

**衝突規則**：高層級贏。即使你在 CLAUDE.md 或對話中要求違反 System Prompt 的事，模型也不會照做。

**實務意義**：理解這個層級，就知道該把什麼規則寫在哪裡。長期有效的規範寫在 CLAUDE.md（第 2 層），臨時指令寫在對話中（第 3 層）。
