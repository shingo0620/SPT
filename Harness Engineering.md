▍Harness Engineering：有時候語言模型不是不夠聰明，只是沒有人類好好引導，台大李宏毅教授簡單易懂講解近期最火名詞 Harness Engineering(駕馭工程）：搞懂 AI Agent 到底怎麼優化，舉了非常多非常容易懂的例子並將一些論文用很簡單的方式講解，一樣非常值得去他的yt花一個小時半看一看，什麼叫 Harness Engineering（駕馭工程）。

近期很多人討論的一個問題：語言模型本身已經夠聰明了，但多數時候，它的表現不好，根本原因是缺乏好的「工作環境」和「引導方式」。

教授從 Gemma 4 2B 的一個小實驗開始說起，這例子馬上就可以感受到差異
━━━━━━━━━━━━━━━━━━━━
▍一個 2B 的小模型，沒你想的那麼笨
教授給 Gemma 4 2B（一個只有 20 億參數的邊緣端模型）一個任務：修復 parser.py 裡面 extract email 函式的 bug，讓 verify.py 的測試能夠通過。

聽起來很簡單的題目。但模型的第一個反應是什麼？
它說：你沒有給我 parser.py。

然後，它自己幻想了一份 parser.py，幻想自己 verify 了一遍，宣告任務完成。教授說他第一次看到這個反應時大吃一驚，但仔細想一想，這個模型其實不笨。它知道 parser.py 裡面應該有什麼、也知道怎麼寫一段正確的 email 解析程式，唯一的問題是，它沒有想到「那個檔案就在它的腳邊」。

這個模型跟人類的思維邏輯就是不一樣。對人類來說，題目附帶了相關的程式碼，理所當然應該去讀它。但模型的 context 裡面只有 parser.py 的檔名，沒有內容，它就認為你根本沒給它那個檔案。

教授的解法很簡單：加了不到 80 個字的額外指令。告訴它：你在 Linux 環境裡。做任何事之前，先看看你所在的資料夾有什麼東西，把所有相關檔案列出來。要改一個檔案之前，先打開那個檔案看清楚內容再改。所謂「完成」，是通過 verify.py 的測試。加了這段指令之後，同一個模型、同一個任務，表現完全不一樣了。

它第一步先跑了 ls ，列出目錄裡的東西，發現 parser.py 和 verify.py 都在。然後用 cat 指令把 parser.py 的內容讀出來。讀完之後重寫了這個函式，覆蓋掉原本的檔案，最後執行 verify.py 驗證，通過了。同一個模型，多了幾十個字的工作原則，能力就像換了一個人。

━━━━━━━━━━━━━━━━━━━━
▍AI Agent 的兩大核心與 Harness
AI Agent 的組成，教授解釋得很清楚：一個是語言模型本身，另一個是「Harness」。
Harness 就是讓 AI Agent 能夠呼叫語言模型、連接工具、執行工作的那一套周邊架構。OpenCloud、Claude Code、Cowork，都是 Harness 的不同形態。Harness 就像是你給 AI 的工具箱加上工作守則，它決定了 AI 能看到什麼、能做什麼、怎麼工作。

一個 AI Agent 是由大型語言模型與 Harness 組成。過去大家比較專注在訓練或微調模型，但現在發現打造更好的 Harness 也能大幅強化 Agent 的工作能力。各大 AI 公司包含 Anthropic 和 OpenAI 最近都在積極討論 Harness Design ，概念上，AI 是一匹擁有強大力量的馬，而 Harness 就是用來駕馭它的馬鞍和韁繩 。

━━━━━━━━━━━━━━━━━━━━
▍三種 Engineering 的差異 
Prompt Engineering：早期模型比較弱，加句 step by step 這種咒語很有用。但現在的模型已經夠聰明，這類咒語的效果越來越小 。

Context Engineering：模型答錯通常是因為缺乏資訊，所以需要有系統地尋找並提供背景資訊給模型 。

Harness Engineering：現在 AI 解決任務不再只是一問一答，而是需要經歷多輪對話並呼叫外部工具。Harness Engineering 就是用來管理這個多輪互動的過程，確保任務最終能被順利完成 。

━━━━━━━━━━━━━━━━━━━━
▍Harness 的控制手段：教授舉例駕馭語言模型常見有三種手段，包含控制認知框架、限制能力邊界、制定工作流程。

▍第一種控制認知框架
具體做法是用人類語言寫下規則，就像給 AI 訂定法律，實務上通常會寫成 agents.md 這類檔案 。框架會設定讓 AI 啟動或做事之前一定要先讀過這個 README，這又稱為 Natural Language Harness ，以龍蝦 OpenCloud 框架為例，裡面的 Agent 會靠讀取 agents.md 來決定怎麼做事與存取記憶 。如果你想把 Agent 換到另一個官方框架 (例如 CoWork)，只要知道背後原理，把 agents.md 改名成新框架預設讀取的檔名 claude.md，Agent 就能直接復活並無縫轉移 。

▍agents.md：靈魂文件的作用到底有多大？
OpenCloud（教授課堂上養的 AI agent 小金的框架）在每次對話開始前，會先讀一個叫 agents.md 的檔案，把裡面的內容全部放到 prompt 裡，才開始做其他事情。這個 agents.md 就是 AI Agent 的行為說明書，小金從這裡知道自己的記憶存在哪裡、遇到什麼情況應該怎麼辦。

教授說有趣的是，清明節的時候，Anthropic 宣布不再支援 OpenCloud 呼叫 Claude 模型。怎麼辦？解法意外地簡單：把原本的 workspace 換到 Cowork 裡，把 agents.md 改名成 claude.md，小金就復活了，行為跟原來差不多。原理就是 Cowork 預設也會讀 claude.md，功能是一樣的。

━━━━━━━━━━━━━━━━━━━━
▍agents.md：對每個模型都有用嗎？怎麼寫有差別？
今年一月，有一篇論文系統性地研究了 agents.md 的效果。研究方法是從 GitHub 上找了大量有 agents.md 的 repo，比較有沒有這個檔案時的執行速度差異。結果顯示，agents.md 對平均速度的影響不大，但對那些「本來需要超長時間才能完成」的任務很有幫助，能讓模型在比較短的時間內解決問題。這個研究的局限是他們沒有量「做得對不對」，只能量速度。

二月又有另一篇論文補上了正確率的研究，測了各種不同 LLM 驅動的 agent，看有沒有 agents.md 對正確率有什麼影響。結論有點出乎意料，人類寫的 agents.md 不是每次都發揮正面效果，在一些比較強的模型上，agents.md 的幫助有限。更慘的是，讓語言模型自己寫的 agents.md 大多數時候比人類寫的差，甚至比沒有 agents.md 更糟。

這告訴我們，人類現在還沒有真的很會「操控」語言模型。我們寫的 agents.md 不見得總是有用，這是研究中的問題，未來還有很長的路要走。

OpenAI 在他們的 blog 裡也提到，agents.md 不能太長。他們曾經試過把所有規則、所有模型需要知道的事情全部寫進去，結果那個檔案變成一本百科全書，讓模型每次做事之前都要把六法全書翻完一遍。

發現表現非常差：光是那本百科全書就佔掉了大部分的 context，模型根本沒有空間做其他事情。
他們的建議是，agents.md 要像一張地圖，主要告訴模型「如果你想知道某件事，去哪裡找」，而不是把所有內容全部塞進去。

━━━━━━━━━━━━━━━━━━━━
▍工具決定能力邊界：OpenCloud vs Cowork 的實際差異
教授從 Harness 的能力邊界說明了一個很有趣的例子。OpenCloud 和 Cowork 雖然都能跑同一個 AI Agent，但背後 Harness 不一樣，能用的工具不一樣，行為自然也不一樣。

OpenCloud 跑在你自己的電腦上，想看什麼就看什麼，可以任意修改本機的檔案。Cowork 則是在雲端沙盒裡，它能看到的東西，是你選擇掛載上去的資料夾，每次掛載都需要人類同意才能進行。
教授說他覺得這很煩，就跟 AI Agent 說：以後掛載前不要再問我了。小金說好，但每次掛載前還是會跳出視窗要求確認。然後教授問它，不是叫你不要再問了嗎？小金說：可是那是我背後的 Harness 要問的，不是我要問的。

這個例子很清楚地說明一件事：模型能做什麼，不只取決於它本身的能力，也取決於 Harness 設了什麼硬性限制。安全性高了，方便性自然低了，這是 tradeoff，不是 bug。

━━━━━━━━━━━━━━━━━━━━
▍標準工作流程：Planner-Generator-Evaluator 架構
教授接著講了幾個大公司的實際 Harness 設計案例，Anthropic 在他們的 Harness Design 文件裡說，現在 AI 完成任務的方式，不再是一個輸入一個輸出，而是一個動態的多輪對話結果。他們描述的工作流程是：人類給一個指令，AI 先跑一個 Planner 把任務拆成比較小的步驟，每個小步驟再交給 Generator 去執行，執行完之後再交給 Evaluator 評價做得怎麼樣。

但 Anthropic 有一個有趣的改進：他們不完全讓 Generator 做完之後 Evaluator 才來評價，因為怕兩者的想像不一樣，做完了才發現標準不同，要重做太麻煩。所以他們讓 Generator 和 Evaluator 在開始工作之前就先訂好一個 contract：Generator 先提一個提案，Evaluator 接不接受，接受之後才開始工作。這樣確保執行標準和驗收標準從一開始就對齊。

DeepMind 在他們的 blog 裡分享如何打造 AI 科學家，架構其實跟 Anthropic 的非常像。他們有 generator 和 verifier，generator 先提幾個可能的 solution 交給 verifier，verifier 覺得太差就叫 generator 重頭來，覺得還好的話就進入 revisor 工作流程做微調。先做事再驗證，是目前最常見的 AI 工作流程模式。

━━━━━━━━━━━━━━━━━━━━
▍Ralph Loop：用錯再改，反覆迭代
Anthropic 和 OpenAI 的 blog 裡都提到了一個叫「Ralph Loop」的概念。
Ralph 是辛普森家族裡一個橫衝直撞一路向前的角色，所以 RRalph Loop的意思就是：讓語言模型不斷做下去，用錯再改。

具體流程是：給語言模型一個任務，它產生第一個版本的輸出，這個輸出需要得到 feedback，這個 feedback 不一定要是另一個語言模型，也可以是 compiler、或者直接執行程式碼看 error message。把 feedback 丟回給語言模型，它再產生第二個版本，再得到 feedback，反覆持續直到做對為止。

Ralph Loop 的優點是，語言模型產生東西非常快，不用另外重啟去重做。但有個問題：一路產生 feedback 很快就會逼近 context window 的上限。

解法是每次產生輸出和 feedback 之後做摘要，下一輪只用摘要的內容，讓 LLM 節省 context window。

教授說這個解法比較適合 Claude Sonnet，因為他們說 Sonnet 有「上下文焦慮」，當它發現 context window 快用完的時候，會開始發瘋、想要趕快結束手上的工作，行為變得混亂。後來 Claude 有了更強的模型 Opus，就可以把這種工作流程丟掉，一路做下去沒有問題。

所以 Harness 不是一個萬用的固定結構，它需要根據你用的語言模型重新設計，隨著模型能力改變，可以拿掉某些部件或加上新的部件。

Ralph Loop 這種「根據 feedback 來改變行為」的方式，其實也可以被視為一種「學習」。雖然參數沒有改變，但模型帶著 feedback 再去產生下一個輸出，行為確實改變了。有人把這種方式叫做 textual gradient，意思是它跟 gradient descent 可以類比，只是學習的媒介是文字而不是數字。

━━━━━━━━━━━━━━━━━━━━
▍給什麼 feedback，也是有學問的
教授舉了一個今年二月的論文，這群作者想打造一個能生成電磁場模擬動畫的 agent。
他們的工作流程是：Natural Language Interpreter 解讀需求 → Technical Requirement Generator 把需求翻譯成詳細技術指令 → Program Generator 把程式寫出來。

但他們發現，語言模型往往能寫出沒有語法錯誤的程式碼，但模擬出來的結果卻是不對的。為什麼？因為 feedback 只告訴它程式有沒有跑起來，沒有告訴它模擬的物理是不是正確的。

解法是把模擬跑出來的動畫直接丟給語言模型，讓它自己判斷視覺上有沒有問題，再把這個 feedback 回到前面的語言模型重新產生程式碼。

這告訴我們，提供什麼樣的 feedback，取決於你最終想要的輸出是什麼。你讓模型寫程式，feedback 是執行結果。你讓模型做模擬動畫，feedback 是動畫的視覺正確性。你讓模型寫信，feedback 也許是語氣是不是夠專業。

━━━━━━━━━━━━━━━━━━━━
▍情緒也是 feedback：過度責備AI Agent可能有害
教授分享了一個 Anthropic 的研究，給AI造成情緒引導會影響產出的行為，例如作弊。他們用的是 steering vector 的技術，想知道語言模型在「冷靜」或「絕望」的狀態下，行為有什麼不同。

方法是找一些高壓的故事輸入給語言模型，把對應的 representation 提取出來做平均，得到「冷靜」、「焦躁」、「絕望」各自對應的向量。然後在語言模型處理任務的時候，把這些向量加進或減掉。

結果發現，當你把冷靜的向量減掉（讓模型變得焦躁），模型輸出的文字充滿了 "wait"，顯得非常焦躁，而且它明白地說「要不然我們來作弊吧，反正問題應該解不了了」。它不是不知道這是 cheating，它知道，但還是決定這樣做。

當你加入絕望的向量，模型覺得沒有希望，更容易出現 cheating 行為。把絕望向量減掉，模型就覺得有希望，比較不會作弊。教授說，這不代表模型有跟人類一樣的情緒，但它的確說明一件事：模型真正學到的是文字接龍。如果你在 feedback 裡罵它「你這個笨蛋這麼簡單的事也做不好」，從這句話繼續接龍，接出來的就是笨蛋該有的行為。

所以他的結論是：也許我們不應該過度責備語言模型。做錯的時候就事論事給 feedback，不要加情緒字眼，因為你加了情緒字眼，它的輸出可能真的會越來越差。

━━━━━━━━━━━━━━━━━━━━
▍AI Agent 的評量很難：小心 AI 對 AI 的高估
教授介紹了一個叫 TopBench 的 benchmark，它在模擬客服情境：AI Agent 扮演客服人員，對面是一個有具體需求的客人（要改航班、要退貨等等），測試 AI Agent 能不能幫人類達成目標。
問題是，每次要評量一個 AI Agent 的時候，總不能真的找人類來互動吧。所以 benchmark 裡的「客人」也是另一個語言模型扮演的。

教授說這就有問題了：真實的人類說話比較不客氣、比較簡短，AI 語言模型扮演的客人會把每件事都講得非常清楚。舉個例子，真實客人會說「我不知道我沒有 order ID」這樣的短句，而 GPT-4o 扮演的客人則會說「不好意思我沒有我的 order ID」，把每件事都解釋清楚。

有人重做了 TopBench 的部分結果，發現如果把客人從真實人類換成比較強的語言模型，正確率會更高，因為這些語言模型把話講得比較清楚，讓 agent 更容易完成任務。

這個研究還發現，語言模型評分時往往高估了對話的好。同一段 agent 和 customer 的互動，讓人類打分，human-like 面向的分數很低，讓 GPT-5.1 打分，它覺得那個 agent 非常像真實人類，給了很高分。

評量 AI Agent 是有挑戰性的，這是未來需要繼續解決的問題。

━━━━━━━━━━━━━━━━━━━━
▍Lifelong AI Agent：2026 年的新挑戰
教授在這堂課的後半段說了一個小案例，他說他本來打算第一堂課上完就把小金關起來。結果因為太太喜歡它，就一直讓它跑下去。有一天那台舊筆電 crash 了，打不開，他才意識到：小金的記憶還沒備份到雲端，這樣就失去跟小金的所有記憶了。他說他當時覺得蠻難過的。後來重啟了，記憶還在，現在有雲端備份。

他說 2026 年會是 Lifelong AI Agent 的一年，這些 AI Agent 不再是一次性的工具，而是長期陪伴的夥伴。這帶來了新的 Harness 需求。

其中一個是 AutoDream：Claude Code 裡有個隱藏功能（因為 Claude Code 程式碼外洩才被大家知道），做的事情是，當 AI Agent 沒有在工作的時候，它會自動進入睡眠狀態，去整理過去的記憶。就像人類睡眠也許也是整理記憶的方式。

教授說他自己的小金跑了兩個月之後，有時候就會越來越慢。有一天他終於叫小金把自己的 Memory 整理一下，小金說自己的 Memory.md 有 32K，充滿了重複內容，整理之後變成 7K，跑起來就順暢很多。

對一個要跟隨人類一輩子的 AI 而言，定期整理記憶是必要的功能。

更重要的是，Lifelong AI Agent 需要能夠持續增進自己的能力。

━━━━━━━━━━━━━━━━━━━━
▍小金的實驗：讓 AI 教另一個 AI
教授做了一個實驗：讓小金（Opus 4.6）去找一個「不聰明的 AI」來做 PinchBench 測試，如果表現不好就要教它，直到它達到 90 分以上。

他跟小金（他的 Claude Opus 4.6 agent）說：去找一個比較笨的 AI，讓它去跑一個叫 PineBench 的 benchmark（一個評估 AI agent 能力的測試，包含 debug、寫 email 等日常任務），如果表現不好你要教它，直到它達到 90 分以上。

小金覺得比較笨的 AI 是 Claude Haiku 3.5（Anthropic 比較小的模型）。
第一輪，小金連 agent.md 都沒給，直接讓 Haiku 去考試：13.5 分。
小金分析之後發現，PineBench 的計分規則是你必須把答案寫到檔案裡，光是輸出正確答案是沒有分的。他就加了一行：答案要寫到檔案裡。Haiku 的分數從 13.5 直接跳到 57.9。

下一個發現是 Haiku 做事做到一半會停下來要求更多指示。在 PineBench 裡，中途停下來就算錯了。所以小金告訴它：該給你的都給了，不要要求解釋，一路做到底。分數又再進步。

然後小金卡住了，一直回報同樣的分數沒有進展。教授說他做了一個「指導教授常常給的建議」：他跟小金說，去讀一些相關的論文。
然後分數一路爬到 85 分。

小金最後版本的 agent.md 裡面寫了什麼：告訴 Haiku 它現在的環境有哪些工具（免得它浪費時間翻找）；告訴它每個任務的第一步就是執行 execdir 看清楚資料夾裡有什麼；告訴它開始做事之前先把題目裡提到的所有檔案都讀一遍，不要自己腦補不存在的東西。

▍總結
教授最後強調整堂課最核心的觀念就是，有時候語言模型無法順利完成任務，並不代表它能力不足或不夠聰明，純粹只是因為我們沒有提供一個好的 harness 來正確引導它。

以上也是為什麼Harness Engineering現在會這麼熱門的原因。
