一種使用 LLM 建立個人知識庫的模式。

這是一份構想文件，設計上是讓你可以直接複製貼上到你自己的 LLM Agent 中使用（例如 OpenAI Codex、Claude Code、OpenCode / Pi 等等）。它的目標是傳達高層次的想法，而具體細節則由你的 agent 與你協作完成。

## 核心概念

大多數人使用 LLM 搭配文件時，體驗通常像是 RAG：你上傳一批檔案，LLM 在查詢時擷取相關片段，然後生成答案。這樣可行，但 LLM 每次回答問題時，其實都在從頭重新發現知識，沒有累積性。若你提出一個需要綜合五份文件才能回答的細微問題，LLM 每次都得重新找出並拼湊相關片段。沒有任何東西真正被建立起來。

NotebookLM、ChatGPT 的檔案上傳功能，以及多數 RAG 系統，基本上都是這樣運作的。這裡的想法不同。不是在查詢時才從原始文件中擷取內容，而是讓 LLM **逐步建立並維護一個持久存在的 wiki** —— 一組結構化、彼此連結的 Markdown 檔案，位在你與原始資料之間。當你加入新資料來源時，LLM 不只是為了日後檢索而建立索引。

它會閱讀資料、萃取重點資訊，並把內容整合進既有 wiki 中 —— 更新實體頁面、修訂主題摘要、標註新資料與舊主張的矛盾之處，並強化或挑戰正在演化的綜合結論。知識只需要被整理編譯一次，之後持續保持最新，而不是每次查詢都重新推導。這才是關鍵差異：**wiki 是一個持續存在、會不斷複利增長的成果物。** 交叉引用已經存在，矛盾也早已被標記出來。

綜整內容已經反映你讀過的一切。每新增一份來源、每提出一個問題，wiki 都會變得更豐富。你幾乎不需要自己撰寫 wiki —— 全部都由 LLM 來寫與維護。你負責的是找資料、探索、提出正確問題。LLM 則處理那些真正讓知識庫長期有用的繁瑣工作 —— 摘要、交叉引用、歸檔、整理與維護。

實際上，我會在一邊開著 LLM agent，另一邊開著 Obsidian。LLM 會根據我們的對話直接編輯內容，而我可以即時瀏覽結果 —— 點連結、查看圖譜視圖、閱讀更新後的頁面。Obsidian 是 IDE；LLM 是程式設計師；wiki 是程式碼庫。這個模式可以應用在許多不同情境。

幾個例子：

- **個人用途**：追蹤自己的目標、健康、心理狀態、自我成長 —— 將日記、文章、Podcast 筆記整理歸檔，隨時間建立出一幅結構化的自我圖像。
- **研究**：在幾週或幾個月內深入鑽研某個主題 —— 閱讀論文、文章、報告，逐步建立一份完整 wiki，並讓核心論點持續演化。
- **讀一本書**：隨著閱讀過程整理每一章，建立角色、主題、情節線與它們之間關聯的頁面。

到最後，你會得到一份內容豐富的伴讀 wiki。可以想像《魔戒》粉絲 wiki 像是 [Tolkien Gateway] —— 數千個彼此相連的頁面，涵蓋角色、地點、事件、語言，由志工社群多年來共同建立。你可以在閱讀時，個人化地打造一套類似的系統，並讓 LLM 負責所有交叉引用與維護工作。

- **商業／團隊用途**：由 LLM 維護的內部 wiki，資料來自 Slack 討論串、會議逐字稿、專案文件、客戶通話紀錄。也可以讓人類參與審核更新。wiki 能保持最新，因為那些團隊成員都不想做的維護工作，交給了 LLM。
- **競品分析、盡職調查、旅遊規劃、課程筆記、興趣深挖** —— 任何你會隨時間累積知識、而且希望它們被妥善組織而不是四散各處的情境，都適用。

## 架構

整體有三層：

**原始資料（Raw sources）** —— 你精心挑選的一批來源文件，例如文章、論文、圖片、資料檔案。這些資料是不可變的 —— LLM 只會讀它們，不會修改它們。這是你的真實來源。

**wiki** —— 一個由 LLM 生成的 Markdown 檔案目錄。內容可能包含摘要、實體頁、概念頁、比較頁、總覽頁、綜整頁。這一層完全由 LLM 負責。

它會建立頁面，在新來源加入時更新它們，維護交叉引用，並確保整體一致性。你負責閱讀；LLM 負責寫作。

**schema** —— 一份文件（例如 Claude Code 使用的 `CLAUDE.md`，或 Codex 使用的 `AGENTS.md`），告訴 LLM 這個 wiki 的結構、慣例，以及在匯入資料、回答問題、維護 wiki 時應遵循哪些工作流程。

這是最重要的設定檔 —— 它讓 LLM 成為一個有紀律的 wiki 維護者，而不是一個泛用聊天機器人。隨著你逐漸找出哪些做法適合你的領域，你與 LLM 會一起演化這份設定。

## 操作方式

**匯入（Ingest）**  
你把新的資料來源放進原始資料集合，然後叫 LLM 去處理它。

一個典型流程是：LLM 讀取來源，和你討論關鍵重點，在 wiki 中寫一頁摘要，更新索引，更新 wiki 中相關的實體頁與概念頁，最後在日誌中追加一筆紀錄。單一資料來源可能會影響 10 到 15 個 wiki 頁面。就我個人而言，我偏好一次匯入一個來源，並且持續參與 —— 我會閱讀摘要、檢查更新內容，並指導 LLM 強調哪些地方。不過你也可以一次批次匯入很多來源，只是監督程度會較低。

採用哪種流程，取決於你自己。你應該發展出最適合你風格的工作方式，並把它寫進 schema 裡，供未來的工作階段沿用。

**查詢（Query）**  
你針對 wiki 提問。LLM 搜尋相關頁面、閱讀它們，然後整理出附帶引用的回答。回答形式可以依照問題不同而變化 —— 可能是一頁 Markdown、比較表格、簡報（Marp）、圖表（matplotlib），甚至是 canvas。

其中一個重要洞見是：**好的回答可以回存到 wiki 裡，變成新的頁面。** 你要求的比較、分析、你發現的新關聯 —— 這些都很有價值，不應該消失在聊天紀錄裡。這樣一來，你的探索也會像匯入來源一樣，在知識庫中產生複利效果。

**檢查（Lint）**  
定期請 LLM 幫 wiki 做健康檢查。

可以檢查的內容包括：頁面之間的矛盾、已被新資料推翻的過時主張、沒有任何反向連結的孤立頁面、被提到但沒有獨立頁面的重要概念、缺少的交叉引用、以及可以透過網路搜尋補上的資料缺口。LLM 很擅長建議值得追問的新問題，以及值得尋找的新資料來源。這能讓 wiki 在成長過程中維持健康。

## 索引與日誌

有兩個特殊檔案，能幫助 LLM（以及你）在 wiki 成長後依然容易導航。

它們的用途不同：

**`index.md`** 側重內容。它像是 wiki 的總目錄 —— 列出每一頁，附上連結、一行摘要，以及可選的中介資料，例如日期或來源數量。內容可依類別組織（實體、概念、來源等等）。LLM 每次匯入都會更新它。回答問題時，LLM 會先讀索引找出相關頁面，再深入閱讀那些頁面。

在中等規模下（大約 100 個來源、數百頁內容），這種方式意外地非常有效，也避免了建立嵌入式 RAG 基礎設施的需求。

**`log.md`** 則側重時間順序。它是一份只能追加、不能覆寫的紀錄，記下發生過什麼事以及何時發生 —— 匯入、查詢、lint 檢查等等。有個實用技巧：如果每則紀錄都以一致前綴開頭（例如 `## [2026-04-02] ingest | Article Title`），那麼日誌就可以用簡單的 Unix 工具解析 —— 例如 `grep "^## \[" log.md | tail -5` 就能取出最近五筆紀錄。

日誌讓你看見 wiki 如何隨時間演化，也幫助 LLM 理解最近完成了哪些工作。

## 可選項：CLI 工具

到了某個階段，你可能會想打造一些小工具，幫助 LLM 更有效率地操作 wiki。最直覺的工具就是 wiki 頁面的搜尋引擎 —— 在小規模下，索引檔就夠了；但隨著 wiki 擴大，你會需要更正式的搜尋能力。

[qmd] 是一個不錯的選擇：它是一套針對 Markdown 檔案的本地搜尋引擎，支援混合式 BM25／向量搜尋與 LLM 重排序，而且全部都在本機裝置上執行。它同時提供 CLI（讓 LLM 可以透過 shell 呼叫）與 MCP server（讓 LLM 能以原生工具方式使用）。你也可以自己做更簡單的版本 —— 當需求出現時，LLM 能幫你快速寫出一個基本搜尋腳本。

## 技巧與訣竅

- **Obsidian Web Clipper** 是一個瀏覽器擴充套件，可以把網頁文章轉成 Markdown。  
    它很適合快速把資料來源放進你的原始資料集合。
- **把圖片下載到本機。**  
    在 Obsidian 的 Settings → Files and links 中，把「Attachment folder path」設為固定資料夾（例如 `raw/assets/`）。接著到 Settings → Hotkeys，搜尋「Download」，找到「Download attachments for current file」，並幫它綁定快捷鍵（例如 Ctrl+Shift+D）。當你剪藏完一篇文章後，按下快捷鍵，所有圖片就會被下載到本機磁碟。
    
    這不是必要的，但很有用 —— 它讓 LLM 能直接查看並引用圖片，而不是依賴可能失效的網址。要注意的是，LLM 目前無法原生地在一次處理中同時完整讀取含有內嵌圖片的 Markdown。可行的做法是先讓 LLM 讀文字內容，再另外查看部分或全部被引用的圖片，以獲得額外上下文。雖然有點笨拙，但已經足夠實用。
    
- **Obsidian 的 graph view** 是觀察 wiki 結構最好的方式 —— 哪些頁面互相連結、哪些頁面是樞紐、哪些是孤點。
- **Marp** 是一種以 Markdown 為基礎的簡報格式，Obsidian 有對應外掛。很適合直接從 wiki 內容生成簡報。
- **Dataview** 是 Obsidian 的一個外掛，可以對頁面的 frontmatter 執行查詢。如果你的 LLM 會在 wiki 頁面中加入 YAML frontmatter（例如標籤、日期、來源數量），Dataview 就能動態產生表格與清單。
- 這整個 wiki 其實就只是由 Markdown 檔案組成的 git repo。你可以免費獲得版本歷史、分支與協作能力。

## 為什麼這方法有效

維護知識庫最麻煩的部分，不是閱讀，也不是思考 —— 而是帳務式的整理工作。更新交叉引用、讓摘要保持最新、標記新資料何時推翻舊主張、在數十個頁面之間維持一致性。人們之所以放棄 wiki，是因為維護成本的增長速度，比 wiki 帶來的價值還快。

LLM 不會覺得無聊，不會忘記更新交叉引用，也能一次處理 15 個檔案。wiki 之所以能持續維護，是因為維護成本接近零。人類的工作是挑選資料來源、引導分析、提出好問題，並思考這些內容真正代表什麼。LLM 的工作則是其餘的一切。

這個想法在精神上與 Vannevar Bush 於 1945 年提出的 Memex 有關 —— 一種個人化、經過策展的知識儲存系統，文件之間有聯想式的路徑相連。

Bush 的願景其實比後來的網路更接近這裡描述的東西：私人的、主動策展的，而且文件之間的關聯與文件本身同樣有價值。他當年無法解決的問題是：誰來做維護？現在這部分可以交給 LLM。

## 備註

這份文件刻意保持抽象。它描述的是一種想法，而不是某個特定實作。

實際的目錄結構、schema 慣例、頁面格式、工具鏈 —— 這些都會取決於你的領域、你的偏好，以及你使用的 LLM。上面提到的一切都是可選且模組化的 —— 有用的就採用，沒用的就忽略。

例如：你的來源可能全部都是純文字，那你根本不需要處理圖片。你的 wiki 也許夠小，只靠索引檔就足夠，不需要搜尋引擎。你可能也不在意簡報生成功能，只想要 Markdown 頁面。
