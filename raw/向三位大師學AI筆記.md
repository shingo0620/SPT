自從Andrej Karpathy推出他的LLM Wiki之後，又有很多大師開始分享他們的筆記，整理如下

完整比較放在留言連結，有興趣的可以去看看，什麼樣的AI筆記系統適合你

當然不免俗的，我也拿來和我的系統進行比較，然後AI正在修改我的系統，一起像大師學習

---------

第一位大師：Andrej Karpathy

他是誰

Andrej Karpathy 是當代深度學習與生成式 AI 領域最具辨識度的實作者與教育者之一。史丹佛大學（Stanford）博士，師承視覺與學習研究脈絡；早年以經典公開課 CS231n（卷積神經網路與視覺辨識，Convolutional Neural Networks for Visual Recognition）影響整代工程師如何入門深度學習。職涯上，他是 OpenAI 創立初期的研究科學家之一，後出任 Tesla 的 AI 總監（Director of AI），主導 Autopilot 相關的神經網路與電腦視覺，把研究推進量產車用系統；近年亦再度與 OpenAI 深度合作，並以長文、演講與 「Software 2.0」「vibe coding」 等概念，塑造開發者如何理解大型語言模型與 Agent。2024 年創辦 Eureka Labs，往 AI 原生的教育產品延伸。

換句話說：當他寫下「LLM Wiki」這類個人知識庫模式，背後不只是一篇 Gist——而是從學術、自動駕駛到 LLM 應用一以貫之的路徑：把複雜系統講清楚，並抽象成他人能複用的模式；這與「好檢索不如好編纂」的論點，是同一種思維習慣。

他教什麼

他提出的不是一套軟體，而是一種工作模式：用不可變的原始資料當地基，用 LLM 維護一層持久性 wiki（摘要、實體、概念、交叉引用），再用一份 schema（如 CLAUDE.md／AGENTS.md）約束「怎麼攝入、怎麼查詢、怎麼健檢」。核心洞見是：好檢索不如好編纂——知識要在多次對話與多份來源中被寫進結構，而不是每次都從零拼湊。

最適合誰、什麼情境

個人研究者、長期主題深讀、讀書與論文綜述。

規模尚可控時（例如數十到數百頁），靠 index.md 與連結就能導航，不必急著上向量庫。

願意與 LLM 共寫規則、共演目錄的人——重點是策展與提問，把「維護成本」交給模型。

一句話

Karpathy 給的是羅盤：你的知識庫應該像 wiki 一樣會長大，而不是像上傳檔案一樣被反覆檢索。

第二位大師：Tobi Lütke 

他是誰

Tobi Lütke 是 Shopify 的共同創辦人與長期 CEO，以把複雜商業與工程問題收斂成可擴展系統著稱。他主導的 QMD（Query Markup Documents） 體現同一種品味：工具要強、要本地、要能被 Agent 當作原生能力呼叫。

他教什麼

QMD 是一套裝在你電腦裡的混合搜尋引擎：BM25、向量語意、查詢擴展與重排序，盡量在本地以 GGUF 模型完成。它不替你寫 wiki，但能在你的 Markdown／程式碼／會議記錄上，做到接近當前「檢索管線」的一流品質。重點是：隱私、零雲端 API、可腳本化與 MCP 整合。

最適合誰、什麼情境

工程師與重度筆記使用者：要在離線或敏感資料上搜尋整個 corpus。

需要 AST 感知分塊的程式碼庫與技術文件。

已經有大量檔案、但還沒決定要不要上雲端向量庫的人。

一句話

Tobi 給的是望遠鏡：在資料已存在的前提下，把「找得到」做到極致，而且不離開你的機器。

第三位大師：Garry Tan 

他是誰

Garry Tan 是 Y Combinator 的總裁，也是創投與創業圈的高頻連結者。他的 GBrain 不是示範專案，而是真實在跑的日常系統：萬級 Markdown、千級人物檔、多年日曆與會議轉錄，最後必須用 Postgres + pgvector 與混合搜尋支撐規模。

他教什麼

GBrain 把「知識庫」升級成可同步、可索引、可被 Agent 讀寫的平台：

頁面模型是 compiled truth（當前最佳論述）+ timeline（只增不刪的證據鏈）；

檢索是 關鍵字 + 向量 + RRF 的工程級組合；

Agent 透過 skills 知道何時讀、何時寫、如何做 enrichment 與夢境維護。

這裡的關鍵不是「會搜尋」，而是知識會隨訊號進場而更新，並與 OpenClaw／Hermes 類的 agent memory 分工：一個管世界事實，一個管操作偏好。

最適合誰、什麼情境

人脈、公司、交易、會議高密度的知識工作者。

資料量已讓 grep 失效、需要增量同步與圖譜式關聯的人。

願意承擔雲端資料庫與嵌入成本、換取「長期可運營」的人。

一句話

Garry 給的是營運系統：當個人知識庫像一家公司一樣有資產負債表，你需要的是資料庫與管線，而不只是資料夾。

與Roger's Letter系統比較

我們的系統是什麼

本專案是以投資研究為核心的 AI Agent 系統：在 Cursor 中以 skills 組織研究管線（產業地圖、公司深度、報告與事實檢視等），並以 Web 儀表板呈現公司、產業地圖、報告與內容庫，資料落地於專案內的結構化檔案與資料庫式 JSON，並可與外部平台（例如譯讀筆記同步）銜接。

它不是泛用的「個人 wiki 產生器」，而是領域明確的第二層大腦：投資與研究工作的數位孿生。

與三位大師的對齊

對 Karpathy：我們同樣依賴 可重複的規約與技能（skills），讓「怎麼研究、怎麼寫、怎麼存檔」成為可繼承的資產，而不是一次性對話。研究產出（報告、公司檔、策略頁）即是可複利的 wiki 式沉澱，只是主題鎖在投資與產業。

對 Tobi：我們未必內建一套通用 QMD；若未來要在本機對 data/ 與 Markdown 報告做極致檢索，QMD 是很自然的外掛望遠鏡。本系統強項在研究動作與儀表板，而非取代全域本地搜尋引擎。

對 Garry：我們與 GBrain 最有共鳴的是「實體導向」與「時間演進」：公司以頁面／記錄存在，研究與事件沿時間累積；差別在於我們不以通用社交圖譜與萬頁個人生活史為預設範圍，而是以投資標的、產業與決策為邊界。若規模與檢索複雜度再升級，GBrain 類的 Postgres 混合索引可成為參考的演進方向，而非第一天必備。

一句話對照

Karpathy 問：你的知識有沒有在長？——我們答：有，長在公司與產業的研究資產裡。

Tobi 問：你找不找得到？——我們答：儀表板與 API 內建結構化查找；若要「全本機、全檔案、極致檢索」，再加 QMD。

Garry 問：你能不能每天自動變聰明？——我們答：skills + 管線 + 同步讓研究流程複利；生活全域與超大規模向量庫則非本系統預設戰場。
